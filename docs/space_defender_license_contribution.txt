4. CONTRIBUTION GUIDELINES
------------------------------------------------------------

Contributions are welcome from the open source community.

Before contributing:

1. Fork the repository
2. Create a new feature branch
3. Implement the feature or bug fix
4. Test your changes
5. Submit a pull request

------------------------------------------------------------
5. BRANCH STRUCTURE
------------------------------------------------------------

The project SHALL use the following Git branch structure.

main
Contains stable release versions of the project.

development
Contains integration of completed features before release.

feature branches
Created for specific features or bug fixes.

Example feature branch names:

feature/enemy-ai
feature/boss-system
fix/collision-bug

------------------------------------------------------------
6. COMMIT MESSAGE FORMAT
------------------------------------------------------------

All commits MUST follow a structured format.

Format:

type: short description

Examples:

feat: add enemy spawning system
fix: resolve collision detection issue
refactor: improve weapon architecture
docs: update project documentation

Commit Types:

feat
fix
refactor
docs
test
chore

------------------------------------------------------------
7. PULL REQUEST RULES
------------------------------------------------------------

Pull requests MUST include:

Clear description of changes
Reference to related issues
Testing confirmation
Code following project coding standards

Pull requests will be reviewed before merging.

------------------------------------------------------------
8. CODE OWNERSHIP
------------------------------------------------------------

The project maintainer has final authority over:

Pull request approvals
Architecture decisions
Release versions

Major architectural changes require discussion before implementation.

------------------------------------------------------------
9. VERSIONING STRATEGY
------------------------------------------------------------

The project SHALL follow Semantic Versioning.

Version format:

MAJOR.MINOR.PATCH

Examples:

1.0.0 Initial release
1.1.0 New gameplay features
1.1.1 Bug fixes

Version increments:

MAJOR: incompatible changes
MINOR: new features
PATCH: bug fixes

------------------------------------------------------------
10. RELEASE PROCESS
------------------------------------------------------------

The release process SHALL include:

Completion of development milestones
Successful testing phase
Code review completion
Tagging release version
Publishing release notes
